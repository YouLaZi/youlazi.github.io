向量数据库快速入门

从 Embedding 到相似度搜索，面向 Java 开发者的向量数据库实战指南。

一、为什么需要向量数据库

传统数据库按精确值查询（WHERE name = ‘张三’），但 AI 需要按语义相似度查询（”和这句话意思最接近的内容”）。

维度 传统数据库 向量数据库
查询方式 精确匹配 / SQL 最近邻搜索（ANN）
数据类型 结构化数据 高维向量（通常 768-1536 维）
典型场景 用户信息、订单 语义搜索、推荐、RAG
代表 MySQL、PostgreSQL Chroma、Milvus、pgvector

二、Embedding（嵌入）原理

将文本转化为固定维度的浮点数数组。语义相近的文本，向量距离越近。

核心直觉：Embedding 就是把”意思”变成”坐标”。意思相近的词，在多维空间里距离近。

三、相似度算法

算法 公式 范围 适用场景
余弦相似度 cos(θ) = A·B / (|A|×|B|) [-1, 1] 文本语义搜索（最常用）
欧氏距离 L2 = √Σ(ai-bi)² [0, ∞) 图像特征匹配
内积 IP = Σ(ai×bi) (-∞, ∞) 向量已归一化时等价余弦

四、Chroma（轻量级首选）

最适合快速原型开发，零配置，Python 原生。

五、pgvector（PostgreSQL 扩展）

适合已有 PostgreSQL 的项目，无需额外基础设施。

HNSW vs IVFFlat

维度 HNSW IVFFlat
查询速度 快 中等
构建速度 慢 快（需训练）
内存占用 大 小
精确度 高 需调参
推荐场景 数据量 < 100万 数据量大，内存有限

六、Milvus（分布式生产级）

适合大规模生产环境，支持百亿级向量。

八、最佳实践

  1. 选 Embedding 模型：text-embedding-3-small（性价比）或 bge-large-zh（中文优化，免费本地部署）
  2. 向量维度：768（轻量）或 1536（标准），越高精度越好但越慢
  3. 索引选择：数据量 < 100万用 HNSW，更大考虑 IVFFlat 或 Milvus
  4. 归一化：使用余弦相似度前确保向量已归一化
  5. 混合检索：向量搜索 + 关键词搜索（BM25）组合效果最佳

九、常见问题

Q: 为什么向量搜索结果不太准？
A: 检查 Embedding 模型是否适合你的语言/领域；调大 chunk_size；增加 top-k 数量；用 Reranker 重排。

Q: pgvector 和独立向量库怎么选？
A: 团队已有 PG 且数据量不大 → pgvector；需要高性能/大规模 → Milvus；快速验证想法 → Chroma。

Q: 向量数据库需要 GPU 吗？
A: 查询不需要（CPU 即可），训练/构建索引时 GPU 加速明显。
