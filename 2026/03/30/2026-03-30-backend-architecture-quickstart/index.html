二、Spring Boot 3.x

自动配置原理

注解速查

注解 作用
@RestController @Controller + @ResponseBody
@Service 业务层，标记 Bean
@Autowired / @RequiredArgsConstructor 依赖注入（推荐后者）
@ConfigurationProperties 绑定配置文件前缀
@Transactional 声明式事务

Starter 机制

spring-boot-starter-web 自动引入：spring-webmvc、tomcat、jackson 等。

三、Spring Cloud 微服务

Nacos 注册中心

服务启动后自动注册到 Nacos，其他服务通过服务名调用。

OpenFeign 远程调用

Gateway 网关

类比：Gateway 就是 Nginx，但它是 Java 写的，能感知服务注册中心。

四、MyBatis-Plus

CRUD

代码生成器

五、MySQL

索引优化

事务隔离级别

级别 脏读 不可重复读 幻读 性能
Read Uncommitted ❌ ❌ ❌ 最高
Read Committed ✅ ❌ ❌ 高
Repeatable Read（默认） ✅ ✅ ❌（InnoDB 基本解决） 中
Serializable ✅ ✅ ✅ 最低

主从复制原理

作用：读写分离（主写从读）、数据备份、故障切换。

六、Redis

数据结构

类型 命令 典型场景
String SET/GET 缓存、分布式锁
Hash HSET/HGET 对象存储（用户信息）
List LPUSH/RPOP 消息队列
Set SADD/SMEMBERS 去重、共同好友
ZSet ZADD/ZRANGE 排行榜

持久化

方式 原理 优缺点
RDB 定时快照（二进制） 恢复快，可能丢数据
AOF 记录每条写命令 数据安全，文件大

推荐：RDB + AOF 混合（Redis 4.0+）。

Sentinel 哨兵

常见应用场景

七、Elasticsearch

倒排索引原理

MySQL：通过 ID 找内容（正排）
ES：通过内容找 ID（倒排）

DSL 查询

ES vs MySQL

维度 MySQL Elasticsearch
擅长 精确查询、事务 全文搜索、模糊匹配
分页 LIMIT 深分页慢 深分页也快（scroll）
扩展 主从 分布式天然扩展
适合 业务数据 日志、搜索、分析

八、RabbitMQ vs Kafka

维度 RabbitMQ Kafka
模型 消息队列（消费后删除） 消息流（可重复消费）
吞吐量 万级 百万级
延迟 微秒级 毫秒级
持久化 支持但影响性能 天然持久化（磁盘）
适用 业务消息（订单、通知） 日志、大数据流
消费模型 推/拉 拉

选型建议：

  • 业务系统消息通知 → RabbitMQ
  • 日志收集、大数据 → Kafka
  • 不确定 → 先 RabbitMQ（学习成本低）

九、Python FastAPI（与 Spring Boot 对比）

维度 Spring Boot FastAPI
语言 Java Python
类型安全 编译期 运行期（Pydantic）
异步 需配置 原生 async/await
性能 高 中高（比 Flask 快很多）
适合 大型业务系统 AI API、快速原型

十、Go 基础（与 Java 并发对比）

Goroutine vs Thread

维度 Java Thread Go Goroutine
内存 ~1MB/线程 ~2KB/goroutine
创建开销 大 极小
调度 OS 内核调度 Go 运行时调度（M:N）
适合 CPU 密集 IO 密集

Channel（通信）

十一、最佳实践

  1. Java 新项目：用 Java 17 + Spring Boot 3.x + MyBatis-Plus
  2. 缓存策略：Cache Aside（先查缓存 → miss 查 DB → 写缓存）
  3. 分布式锁：Redis SET NX EX + Lua 释放
  4. 消息队列：消费端做好幂等（用唯一 ID 去重）
  5. SQL 优化：EXPLAIN 看执行计划，复合索引遵循最左前缀
  6. 微服务：不要一开始就拆，单体够用时别过度设计

十二、常见问题

Q: Spring Boot 和 Spring Cloud 什么关系？
A: Spring Boot 做单体应用，Spring Cloud 把多个 Spring Boot 应用组成微服务集群。

Q: Redis 缓存穿透怎么办？
A: 布隆过滤器（挡非法请求）+ 缓存空值（设短过期时间）。

Q: 什么时候该拆微服务？
A: 团队 > 10人、模块边界清晰、需要独立部署/扩缩容时再拆。过早微服务化是灾难。
