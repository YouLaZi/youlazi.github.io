IdeaFlow - 企业级RAG知识库问答系统 技术文档

1. 项目概述

项目背景

企业在日常运营中积累了大量文档（PDF、Word、Markdown等），但传统搜索方式难以从海量非结构化数据中精准获取答案。本项目基于RAG（Retrieval-Augmented Generation）技术，构建企业级知识库问答系统，实现”上传文档→智能问答”的全链路能力。

项目目标

  • 实现RAG全链路：文档解析→向量化→检索→生成
  • 基于Quivr开源框架二次开发，打造企业级知识库
  • 支持多模型热切换，降低厂商锁定风险
  • 达到P95响应<2s的生产级性能

项目成果

  • 担任 Tech Lead，主导RAG核心模块开发
  • Top-K召回准确率 85%+
  • P95响应 <2s
  • 支持 10+文档格式
  • 支持 OpenAI / Claude / 本地模型热切换

2. 系统架构

整体架构

基于Quivr二次开发

Quivr是开源RAG框架，提供：

  • quivr-core：核心Brain类，from_files()创建RAG知识库，ask()进行问答
  • YAML配置Workflow：灵活定义检索增强生成流程
  • 多LLM支持：OpenAI、Anthropic、本地模型等
  • 多向量数据库：pgvector、Qdrant、Milvus等

3. 技术栈详解

技术 用途
LangChain LLM编排框架，多轮对话管理，Prompt模板
pgvector PostgreSQL向量扩展，存储文档嵌入向量
HNSW索引 高维向量近似最近邻搜索算法
FastAPI Python异步Web框架，高性能API服务
Next.js React全栈框架，前端SSR/SSG
TypeScript 类型安全的前后端开发
Prisma ORM框架，类型安全的数据库操作
Cohere Reranker 检索结果重排序，提升召回精度
quivr-core RAG核心引擎，Brain类封装

支持的文档格式

PDF, DOCX, TXT, MD, CSV, XLSX, PPTX, HTML, JSON, EPUB, 图片(OCR)（10+种）

4. 核心功能模块

4.1 文档处理模块

  • 文档解析：支持10+格式的文件上传和解析
  • 文本分块：智能Chunk策略（按段落/语义分块，Chunk大小可配置）
  • 向量化：调用Embedding模型生成向量表示
  • 存储入库：向量+元数据写入pgvector

4.2 知识库管理模块

  • 创建/删除知识库（Brain实例）
  • 知识库文件管理（上传/删除/列表）
  • 知识库权限控制
  • 知识库容量监控

4.3 问答模块

  • RAG问答：基于知识库内容的精准回答
  • 多轮对话：LangChain Conversation Chain，维护上下文
  • 来源追溯：返回答案对应的原文片段
  • 混合检索：向量检索 + 关键词检索融合

4.4 Workflow引擎（YAML配置）

5.2 pgvector + HNSW索引优化

  • HNSW（Hierarchical Navigable Small World）：多层图结构，查询时间复杂度接近O(logN)
  • 索引参数调优：m=16, ef_construction=200, ef_search=40
  • 向量维度：取决于Embedding模型（如1536维 for OpenAI ada-002）
  • IVFFlat vs HNSW选择：HNSW在召回率和查询速度间取得更好平衡

6. 数据库设计

核心表结构（PostgreSQL + pgvector）

知识库表 (brains)

字段 类型 说明
id UUID 主键
name VARCHAR(200) 知识库名称
description TEXT 描述
owner_id UUID 所有者
model_config JSONB LLM配置
created_at TIMESTAMP 创建时间

文档表 (files)

字段 类型 说明
id UUID 主键
brain_id UUID 关联知识库
file_name VARCHAR(500) 文件名
file_size BIGINT 文件大小
content TEXT 解析后的文本
chunk_count INT 分块数量
status VARCHAR(20) 处理状态

向量表 (knowledge_vectors)

字段 类型 说明
id UUID 主键
file_id UUID 关联文件
content TEXT 文本片段
embedding vector(1536) 向量（pgvector类型）
metadata JSONB 元数据（页码、章节等）

7. 部署方案

  • 前端：Next.js → Vercel / Docker部署
  • 后端：FastAPI → Docker容器化部署
  • 数据库：PostgreSQL + pgvector → 云数据库/RDS
  • LLM API：外部调用（OpenAI/Claude），本地模型可选Ollama部署
